.1 ما هو الـ Middleware؟ What is Middleware?

الـ Middleware هو دالة تعترض الطلب في منتصف رحلته من الاستقبال حتى الرد — تستطيع قراءة الطلب وتعديله وإيقافه أو تمريره للتالي:

Responseres.json()
🧩 شكل Middleware الأساسي
// دالة middleware = 3 parameters: req, res, next
function myMiddleware(req, res, next) {
  // 1. يمكنك قراءة أو تعديل req وres
  console.log('طلب واصل:', req.method, req.url);

  // 2. إما تستمر في السلسلة
  next();

  // 3. أو ترد وتوقف السلسلة
  // res.status(401).json({ error: 'غير مصرح' });
}

// تطبيق على جميع الطلبات
app.use(myMiddleware);

// تطبيق على route محدد فقط
app.get('/api/secret', myMiddleware, (req, res) => {
  res.json({ data: 'سري' });
});

الطلب كالمسافر — يمر عبر نقاط تفتيش متتالية: فحص التذكرة (Auth) ← تسجيل الأمتعة (Body parsing) ← تسجيل البيانات (Logger) — كل نقطة إما تمرّره أو توقفه.

.2 next() — قلب السلسلة The next() Function

next() هي الدالة التي تنتقل بك للـ middleware التالي — بدونها الطلب يتجمّد:

🔗 سلوك next() في الحالات المختلفة
function middlewareA(req, res, next) {
  console.log('A: بدأ');

  // ✅ تابع للـ middleware التالي
  next();

  console.log('A: انتهى');  // ينفَّذ بعد عودة السلسلة
}

function middlewareB(req, res, next) {
  console.log('B: بدأ');
  next();
  console.log('B: انتهى');
}

app.use(middlewareA);
app.use(middlewareB);
app.get('/', (req, res) => {
  console.log('Handler: يرد');
  res.json({ ok: true });
});

// الترتيب الفعلي:
// A: بدأ → B: بدأ → Handler: يرد → B: انتهى → A: انتهى
⚠️ متى تستدعي next() ومتى لا تستدعيها؟
// ✅ استدعِ next() إذا لم ترد وتريد الاستمرار
function logger(req, res, next) {
  console.log(req.method, req.url);
  next();  // ← مطلوب
}

// ✅ استدعِ next(err) لإرسال الخطأ لـ error handler
function riskyMiddleware(req, res, next) {
  try { doSomething(); next(); }
  catch (err) { next(err); }  // ← يُطلق error handler
}

// ✅ لا تستدعِ next() إذا رددت بنفسك
function auth(req, res, next) {
  if (!req.headers['authorization']) {
    return res.status(401).json({ error: 'مطلوب token' });
    // ← لا next() هنا — الاستجابة اكتملت
  }
  next();
}

// ❌ لا تستدعِ الاثنتين معاً!
// res.json({}) ثم next() = خطأ في الـ headers
القاعدة الذهبية: كل middleware إما يستدعي next() أو يرد بـ res.json() أو res.send() — لا كليهما ولا لا شيء. أي middleware لا يفعل أحدهما سيُجمّد الطلب.

.4 Custom Middleware — Logging Request Logger

أكثر middleware تصنعه بنفسك هو الـ Logger — يُسجّل كل طلب وتفاصيله:

📋 Logger بسيط
function requestLogger(req, res, next) {
  console.log(`${req.method} ${req.url}`);
  next();
}

app.use(requestLogger);

// الناتج:
// GET /api/users
// POST /api/users
📊 Logger متقدم مع الوقت والـ Status
function requestLogger(req, res, next) {
  const start = Date.now();

  // نستمع لحدث finish — يُطلَق عند اكتمال الرد
  res.on('finish', () => {
    const ms      = Date.now() - start;
    const status  = res.statusCode;
    const color   = status >= 400 ? '\x1b[31m' : '\x1b[32m';  // أحمر/أخضر
    const reset   = '\x1b[0m';

    console.log(
      `${color}${status}${reset}`,
      req.method.padEnd(6),
      req.url.padEnd(30),
      `${ms}ms`
    );
  });

  next();
}

app.use(requestLogger);

// الناتج:
// 200 GET    /api/users                     12ms
// 201 POST   /api/users                     8ms
// 404 GET    /api/unknown                   3ms
لماذا res.on('finish')? لأن الـ status code لا يكون متاحاً حتى يكتمل الرد — إذا طبعناه في بداية الـ middleware سيكون دائماً undefined. حدث finish يُطلَق بعد إرسال الرد كاملاً.

.5 Custom Middleware — Auth Authentication Middleware

الـ Auth Middleware يتحقق من هوية المستخدم قبل السماح له بالوصول للـ route:

🔐 Auth Middleware بسيط
function requireAuth(req, res, next) {
  // استخرج الـ token من الـ header
  const authHeader = req.headers['authorization'];

  // تحقق من وجوده
  if (!authHeader || !authHeader.startsWith('Bearer ')) {
    return res.status(401).json({
      error: 'يجب تسجيل الدخول — أرسل Authorization: Bearer TOKEN',
    });
  }

  const token = authHeader.split(' ')[1];  // "Bearer xyz" → "xyz"

  // في الواقع: تتحقق من JWT هنا (درس لاحق)
  if (token !== 'my-secret-token') {
    return res.status(403).json({ error: 'Token غير صالح' });
  }

  // أضف بيانات المستخدم للـ req — متاحة للـ handler بعدك
  req.user = { id: 1, name: 'أحمد', role: 'admin' };
  next();
}
🛡️ تطبيق Auth على routes مختلفة
// route محدد — requireAuth يُمرَّر كحجة
app.get('/api/profile', requireAuth, (req, res) => {
  res.json({ user: req.user });  // req.user من الـ middleware
});

// جميع routes /api/admin محمية
app.use('/api/admin', requireAuth);
app.get('/api/admin/users', (req, res) => {
  res.json({ users: [], admin: req.user });
});

// تطبيق على router كامل
const adminRouter = require('./routes/admin');
app.use('/api/admin', requireAuth, adminRouter);

// routes عامة — لا تحتاج auth
app.get('/api/products', (req, res) => {
  res.json({ products: [] });  // متاح للجميع
});
req.user في كل مكان: بعد تعيين req.user في الـ middleware، تستطيع قراءته في أي middleware أو handler يأتي بعده في نفس الطلب — الـ req object يُمرَّر عبر السلسلة كاملة.

.6 Error Handling Middleware Error Handler

هذا النوع الخاص له 4 parameters — Express يُميّزه تلقائياً عن الـ middleware العادي:

Route Handler
أو Middleware
→
throw new Error()
أو next(err)
→
يتخطى كل
middleware عادي
→
Error Handler
(4 params)
💥 Error Handler — 4 Parameters
// يجب أن يكون له بالضبط 4 params — حتى لو لم تستخدم next
function errorHandler(err, req, res, next) {
  // طبع الخطأ في الـ console (للمطور)
  console.error('❌ Error:', err.message);

  const statusCode = err.statusCode || 500;
  const message    = err.message    || 'خطأ داخلي في الخادم';

  res.status(statusCode).json({
    success: false,
    error:   message,
    // لا ترسل stack trace في الإنتاج!
    ...(process.env.NODE_ENV !== 'production' && { stack: err.stack }),
  });
}

// يُوضع آخر شيء بعد جميع الـ routes
app.use(errorHandler);
⚡ إطلاق الـ Error Handler بـ next(err)
// خطأ مخصص مع statusCode
class AppError extends Error {
  constructor(message, statusCode) {
    super(message);
    this.statusCode = statusCode;
  }
}

// في الـ route
app.get('/api/users/:id', async (req, res, next) => {
  try {
    const user = await findUser(req.params.id);

    if (!user) {
      throw new AppError('المستخدم غير موجود', 404);
    }

    res.json({ data: user });
  } catch (err) {
    next(err);  // ← يُرسله للـ error handler
  }
});

// أو بدون try/catch في Express 5 (يُمسك async errors تلقائياً)
// app.get('/api/users/:id', async (req, res) => { throw ... });
Express لا يُمسك async errors تلقائياً في v4: في Express 4 يجب استخدام try/catch + next(err). في Express 5 (الأحدث) يُمسك الأخطاء غير المعالجة من async functions تلقائياً.

const express    = require('express');

const app = express();

// 1. Helmet — أمان (أول شيء)

// 2. CORS — السماح للفرونت إند بالوصول

  origin: ['http://localhost:3000', 'https://mysite.com'],

// 5. Body parsing
app.use(express.json());

// Middleware جاهز بسطرين بدل 50!

.8 سلسلة Middleware كاملة في العمل Full Middleware Chain

تطبيق Express كامل مع Logger + Auth + Error Handler وكل شيء بالترتيب الصحيح:

🛡️ Express — سلسلة Middleware كاملة app.js 
// app.js — الترتيب مهم جداً!
const express = require('express');
const app = express();

// ── 1. Logger (أول شيء) ──────
app.use((req, res, next) => {
  const s = Date.now();
  res.on('finish', () =>
    console.log(
      req.method, req.url,
      res.statusCode,
      `${Date.now()-s}ms`
    )
  );
  next();
});

// ── 2. Body Parsing ──────────
app.use(express.json());

// ── 3. Routes عامة ──────────
app.get('/api/public',
  (req, res) =>
    res.json({data:'للجميع'})
);

// ── 4. Auth Middleware ───────
function requireAuth(req,res,next){
  const h=req.headers.authorization;
  if(!h?.startsWith('Bearer '))
    return res.status(401)
      .json({error:'token مطلوب'});
  req.user={id:1,name:'أحمد'};
  next();
}

// ── 5. Routes محمية ─────────
app.get('/api/profile',
  requireAuth,
  (req,res) =>
    res.json({user:req.user})
);

app.post('/api/data',
  requireAuth,
  (req,res,next) => {
    try{
      if(!req.body.value)
        throw Object.assign(
          new Error('value مطلوب'),
          {statusCode:400}
        );
      res.json({ok:true});
    }catch(e){next(e);}
  }
);

// ── 6. 404 Handler ───────────
app.use((req,res) =>
  res.status(404)
    .json({error:'Not Found'})
);

// ── 7. Error Handler (آخراً) ─
app.use((err,req,res,next) => {
  res.status(err.statusCode||500)
    .json({
      success:false,
      error:err.message
    });
});

app.listen(3000);

📝 التمارين العملية Exercises

تمرين 1: Logger Middleware

اكتب middleware logger يطبع لكل طلب:
[2026-01-01 12:00:00] GET /api/users → 200 (12ms)
مع الوقت والـ status code والمدة.

الحل: 
function requestLogger(req, res, next) {
  const start = Date.now();

  res.on('finish', () => {
    const ms       = Date.now() - start;
    const time     = new Date().toISOString().replace('T', ' ').slice(0, 19);
    const status   = res.statusCode;
    console.log(`[${time}] ${req.method} ${req.url} → ${status} (${ms}ms)`);
  });

  next();
}

app.use(requestLogger);
تمرين 2: API Key Middleware

اكتب middleware يتحقق من وجود header X-API-Key بقيمة معينة. إذا لم يوجد أعد 401، إذا كانت القيمة خاطئة أعد 403.

الحل: 
const VALID_API_KEY = 'my-secret-api-key-2024';

function requireApiKey(req, res, next) {
  const apiKey = req.headers['x-api-key'];

  if (!apiKey) {
    return res.status(401).json({
      error: 'X-API-Key header مطلوب',
    });
  }

  if (apiKey !== VALID_API_KEY) {
    return res.status(403).json({
      error: 'API Key غير صالح',
    });
  }

  next();
}

// الاستخدام
app.use('/api', requireApiKey);
تمرين 3: Validation Middleware

اكتب middleware requireBody(...fields) يتحقق من وجود حقول معينة في req.body. إذا غاب أي حقل أعد 400 مع رسالة تحدد الحقل الناقص.

الحل: 
function requireBody(...fields) {
  return (req, res, next) => {
    for (const field of fields) {
      if (!req.body || req.body[field] === undefined || req.body[field] === '') {
        return res.status(400).json({
          error: `الحقل "${field}" مطلوب`,
        });
      }
    }
    next();
  };
}

// الاستخدام
app.post('/api/users',
  requireBody('name', 'email', 'password'),
  (req, res) => {
    res.status(201).json({ message: 'تم إنشاء المستخدم' });
  }
);
تمرين 4: Error Handler

اكتب errorHandler يُعيد JSON بهذا الشكل:
{ success: false, error: message, statusCode, path: req.url }
بكود الـ error.statusCode أو 500 افتراضياً.

الحل: 
function errorHandler(err, req, res, next) {
  const statusCode = err.statusCode || 500;
  const message    = err.message    || 'خطأ داخلي في الخادم';

  console.error(`[ERROR] ${statusCode}: ${message}`);

  res.status(statusCode).json({
    success:    false,
    error:      message,
    statusCode,
    path:       req.url,
  });
}

// يُوضع آخر شيء في app.js
app.use(errorHandler);
تمرين 5: ترتيب الـ Middleware

رتّب هذه الـ middlewares بالترتيب الصحيح في app.js واشرح لماذا:
أ) app.use(errorHandler)
ب) app.use(express.json())
ج) app.use(requestLogger)
د) routes التطبيق
هـ) app.use((req,res) => res.status(404)...)

الترتيب الصحيح: ج ← ب ← د ← هـ ← أ

• ج (logger) أولاً: يجب أن يُسجّل كل طلب بما فيها المحظورة
• ب (json) ثانياً: يجب تحليل الـ body قبل وصوله للـ routes
• د (routes) ثالثاً: المنطق الأساسي بعد الإعداد
• هـ (404) رابعاً: يلتقط ما لم تلتقطه الـ routes
• أ (error handler) آخراً: يلتقط جميع أخطاء next(err)

اختبر فهمك للـ Middleware — قلب Express!

1كل middleware يجب إما أن يستدعي next() أو يرد بـ res.json() — لا يمكن تركه دون أي منهما.
2ترتيب تعريف الـ middleware في Express لا يؤثر على ترتيب تنفيذها.
3Error Handler Middleware يجب أن يكون له بالضبط 4 parameters: (err, req, res, next).
4يمكن تعيين بيانات في req داخل middleware وقراءتها في الـ route handler الذي يأتي بعده.
5يمكن تطبيق middleware على route واحد فقط بتمريره كحجة: app.get('/path', auth, handler).
