.1 ما هو Git؟ Version Control

Git هو نظام التحكم في الإصدار الأشهر في العالم. يسمح لك بـ:

  • حفظ كل تغيير في كودك بشكل منظم
  • الرجوع لأي نسخة سابقة في أي وقت
  • العمل مع فريق بدون تعارض
  • تجربة ميزات جديدة بأمان (Branches)
🎮 تخيل لعبة فيديو بأنظمة Save:

Git = نظام الـ Save. كل commit = نقطة حفظ. لو حصل أي شيء غلط، ترجع لأي نقطة حفظ سابقة. GitHub = السحابة اللي تحتفظ بنسختك عبر الإنترنت.

Git مقابل GitHub:
GitGitHub
ما هو؟برنامج محليموقع إنترنت
أين يعمل؟على جهازكفي السحابة
الوظيفةتتبع التغييراتمشاركة الكود
بديله—GitLab, Bitbucket

.2 التثبيت والإعداد Setup

1. تثبيت Git:
تحمّل من: git-scm.com
# تأكد من التثبيت
git --version
git version 2.44.0
2. إعداد الهوية (مرة واحدة فقط):
⚙️ الإعداد الأولي
# اسمك (يظهر في كل commit)
git config --global user.name "Ahmed Khalid"

# بريدك (نفس بريد GitHub)

# تأكد من الإعدادات

--global يعني تنطبق على كل مشاريعك. لمشروع واحد فقط، احذف --global.

.3 الأوامر الأساسية Core Commands

دورة حياة الكود مع Git:

Working Directoryتعديل ملفاتك
→
Staging Areagit add
→
Repositorygit commit
→
GitHubgit push
🚀 الأوامر الأساسية
# 1. إنشاء مستودع جديد
git init

# 2. رؤية حالة الملفات
git status

# 3. إضافة ملف للـ Staging
git add index.html
git add .          # إضافة كل الملفات

# 4. حفظ نقطة (Commit)
git commit -m "أضفت الصفحة الرئيسية"

# 5. رؤية سجل التغييرات
git log
git log --oneline  # ملخص سريع
↩️ التراجع والإلغاء
# إلغاء تعديلات ملف (قبل add)
git checkout -- index.html

# إزالة ملف من Staging (بعد add قبل commit)
git restore --staged index.html

# الرجوع لـ commit سابق (للقراءة فقط)
git checkout abc1234

# إلغاء آخر commit (يحافظ على التعديلات)
git reset --soft HEAD~1
الأمرالوظيفة
git initينشئ مستودع Git جديد
git statusيعرض حالة الملفات
git add .يضيف كل التغييرات للـ Staging
git commit -mيحفظ نقطة تغيير مع رسالة
git log --onelineسجل مختصر للـ commits
git diffيعرض التغييرات غير المحفوظة

.4 الفروع (Branches) Branching

الفروع تسمح لك بتطوير ميزات جديدة بأمان بدون التأثير على الكود الرئيسي.

● main: C1 ─── C2 ─── C3 ─────────────── C6 (merge)
└── C4 ─── C5 (feature/login)
🌿 أوامر الفروع
# رؤية الفروع الموجودة
git branch

# إنشاء فرع جديد
git branch feature/login

# الانتقال للفرع
git checkout feature/login

# إنشاء والانتقال معاً (أسرع)
git checkout -b feature/login

# رؤية الفرع الحالي
git branch --show-current

# حذف فرع (بعد الدمج)
git branch -d feature/login
تسمية احترافية للفروع:
feature/اسم-الميزة — ميزة جديدة
fix/وصف-المشكلة — إصلاح خطأ
hotfix/إصلاح-عاجل — إصلاح عاجل في الإنتاج

.5 الدمج وحل التعارضات Merge & Conflicts

🔀 دمج الفروع
# انتقل لـ main أولاً
git checkout main

# ادمج الفرع
git merge feature/login

# حذف الفرع بعد الدمج
git branch -d feature/login
حل تعارض (Conflict):

يحدث التعارض عندما يعدّل شخصان نفس السطر. Git يوقف الدمج ويطلب منك الحل:

⚠️ ملف فيه تعارض
<<<<<<< HEAD (نسختك)
const color = 'blue';
=======
const color = 'red';
>>>>>>> feature/login (النسخة الأخرى)
✅ بعد حل التعارض
# احذف الأسطر <>= واترك الكود الصح
const color = 'blue';

# ثم أضف وأكمل الدمج
git add index.html
git commit -m "حل تعارض: لون الزر"

.6 GitHub والـ Remote GitHub & Remote

☁️ رفع مشروع على GitHub
# 1. أنشئ repo على github.com أولاً
# 2. اربط مشروعك بـ GitHub
git remote add origin https://github.com/username/myproject.git

# 3. ارفع أول مرة
git push -u origin main

# 4. بعد كذا: ارفع بشكل عادي
git push

# سحب التحديثات من GitHub
git pull

# استنساخ مشروع موجود
git clone https://github.com/username/project.git
أوامر Remote مفيدة:
🔗 إدارة الـ Remotes
# رؤية الـ remotes المتصلة
git remote -v

# سحب التحديثات بدون دمج
git fetch origin

# رفع فرع جديد لـ GitHub
git push -u origin feature/login
Pull Request (PR): طلب دمج فرعك في main عبر GitHub. هو الطريقة الاحترافية للمراجعة والموافقة قبل الدمج في الفريق.

.7 سير عمل احترافي Professional Workflow

الطريقة الأشهر في الفرق الاحترافية:

⚡ الـ Feature Branch Workflow
# 1. تأكد من آخر تحديث لـ main
git checkout main
git pull

# 2. أنشئ فرع للميزة
git checkout -b feature/dark-mode

# 3. اشتغل وسجّل التغييرات
git add .
git commit -m "أضفت الوضع الليلي للـ navbar"
git commit -m "أضفت الوضع الليلي للـ footer"

# 4. ارفع الفرع
git push -u origin feature/dark-mode

# 5. افتح Pull Request على GitHub
# 6. بعد الموافقة، ادمج وامسح الفرع
git checkout main
git merge feature/dark-mode
git branch -d feature/dark-mode
git push
ملف .gitignore:
🙈 .gitignore
# ملفات يتجاهلها Git
node_modules/
.env
dist/
*.log
.DS_Store
node_modules يحتوي آلاف الملفات ولا يُرفع أبداً. كل من يستنسخ المشروع يشغّل npm install لتحميلها.

.8 محرر الكود الحي Live Editor

شاهد محاكاة لسجل Git:

🛠️ محاكاة Git Log

📝 التمارين العملية Exercises

تمرين 1: أوامر البداية

اكتب تسلسل الأوامر لـ: إنشاء مستودع جديد، إضافة ملف index.html، ثم حفظه برسالة "أول commit".

الحل: 
git init
git add index.html
git commit -m "أول commit"
تمرين 2: العمل بفروع

أنشئ فرع اسمه feature/contact-page، انتقل إليه، ثم اكتب أمر للتحقق من الفرع الحالي.

الحل: 
git checkout -b feature/contact-page
git branch --show-current
# النتيجة: feature/contact-page
تمرين 3: الربط بـ GitHub

اكتب الأوامر للربط بـ remote repo على GitHub وأول رفع للكود.

الحل: 
git remote add origin https://github.com/username/myproject.git
git push -u origin main
تمرين 4: ملف .gitignore

اكتب محتوى ملف .gitignore يتجاهل: مجلد node_modules، ملف .env، وأي ملف ينتهي بـ .log.

الحل: 
node_modules/
.env
*.log
تمرين 5: سير عمل كامل

ابدأ من main، أنشئ فرع fix/nav-bug، افترض أنك أصلحت الـ navbar وحفظت التغيير، ثم ادمجه في main.

الحل: 
git checkout main
git pull
git checkout -b fix/nav-bug
# (تعديل الملفات)
git add .
git commit -m "إصلاح: مشكلة في الـ navbar"
git checkout main
git merge fix/nav-bug
git branch -d fix/nav-bug
git push

اختبر فهمك لـ Git و GitHub!

1git add يضع الملفات في الـ Staging Area قبل الـ commit.
2GitHub و Git هما نفس الشيء.
3مجلد node_modules يجب وضعه في .gitignore وعدم رفعه.
4git push يسحب آخر التحديثات من GitHub.
5الفروع تسمح بتطوير ميزات جديدة بدون التأثير على main.
